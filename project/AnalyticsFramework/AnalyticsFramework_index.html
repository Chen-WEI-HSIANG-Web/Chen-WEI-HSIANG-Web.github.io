01 / Overview

Why this project exists

The framework was created to solve a practical problem: HR data existed in the system, but insight was still difficult to reuse, compare, and scale across decision scenarios.

Context

In daily HR work, data requests often begin with simple questions such as turnover rate, headcount changes, or new hire status. But over time, the real business need becomes more complex: leaders want to understand what changed, why it changed, and what should be acted on next.

Core Problem
  • Metrics were visible, but interpretation was not always consistent
  • Dashboard logic risked being limited by system guardrails
  • Data domains such as performance, hiring, and learning were still disconnected
  • Leaders needed decision-oriented views, not only static reporting outputs
02 / Framework

How the analytics framework is structured

This project is organized as a layered model so new topics can be added without breaking metric consistency or design logic.

01 —
Workforce
Foundation metrics including headcount, turnover, retention, and workforce structure across region, role, and gender.
02 —
Performance
Rating distribution, calibration outcomes, and future performance-risk insight for business discussions.
03 —
Hiring / Probation
New hire quality, probation pass rate, and early retention analysis to strengthen talent entry decisions.
04 —
Learning
Learning completion, learning history, and future correlation with performance and retention.
Framework Logic
  • Metric governance: definitions are designed to stay stable across future dashboard additions
  • Lifecycle thinking: the model is expanding from workforce visibility to selection, development, and retention
  • System-aware design: implementation respects current Story Report constraints while keeping future migration options open
  • Decision orientation: the long-term target is not only analytics visibility, but business action support

03 / Metrics

Core metrics and implementation logic

The value of this project depends on metric consistency. Each KPI is designed so it can be explained, challenged, and extended over time.

Monthly Turnover Rate

Used to reflect workforce movement in a way that avoids distortion from month-end headcount only.

Monthly Turnover Rate = Monthly Leavers / Average Headcount Average Headcount = (Beginning Headcount + Ending Headcount) / 2
Monthly Retention Rate

Designed to reflect active employee continuity while separating the effect of new hires from retention itself.

Monthly Retention Rate = (Beginning Headcount - Monthly Leavers) / Beginning Headcount
New Hire Retention

Tracks whether employees hired in a defined period remain active after a set time window, such as 3 months.

3-Month New Hire Retention = Active employees hired 3 months ago / Total hires 3 months ago
Implementation Notes
  • Initial build completed in SAP SuccessFactors Story Report
  • Previous nested-calculation guardrail was removed after Product Team discussion, but may return in the future
  • Future architecture still needs evaluation of SAP People Intelligence for long-term scalability
  • Current phase is focused on BU adoption, feedback absorption, and prioritizing next high-value indicators

05 / Impact

What changed because of this project

The short-term output is a dashboard foundation. The longer-term impact is a more scalable way to think about HR data products.

01 —
Clearer Data Language
Metrics were defined with more stable logic so BU discussions can focus less on interpretation gaps and more on action.
02 —
Stronger Expansion Path
The project now has a visible roadmap from workforce reporting into performance, probation, learning, and lifecycle analytics.
03 —
Better Strategic Positioning
This is no longer only a reporting task. It positions HR analytics as an internal product that can support leadership decisions.
06 / Reflection

What this project represents

More than a dashboard build, this project reflects a shift from reporting work toward analytics design and internal product thinking.

Reflection

Building this framework changed the way I think about HR data. I was no longer asking only how to deliver a report, but how to design a model that can scale, survive system limits, and support better decisions over time. It also clarified the direction I want to grow in: work that sits between HR operations, systems, analytics, and long-term internal product design.

This framework can keep growing.

Future additions can include performance calibration insight, probation analysis, learning-to-performance correlation, or a public-safe demo version for portfolio use.
