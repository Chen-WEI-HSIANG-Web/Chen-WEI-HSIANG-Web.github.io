01 / Overview

Why this project existed

The Knowledge Hub was not created as a visual exercise. It was created to solve an operational weakness: important know-how was difficult to retain, transfer, and reuse.

Context

In HR work, many processes are highly dependent on experience. Project logic, system settings, file paths, template usage, and execution rhythm often stay fragmented across folders, people, and memory. That makes onboarding slow and handover risky.

Core Problem
  • Project knowledge depended too heavily on individuals
  • SOPs and templates were hard to locate and connect
  • Operational logic was not visible enough for successors
  • Different HR audiences needed different entry points into the same knowledge base
02 / Design

How I designed the solution

I approached the Hub as an internal product, not just a documentation folder. The goal was to make complex HR knowledge easier to browse, understand, and continue.

Solution Logic
  • Used static HTML/CSS so the site could run in a lightweight internal environment
  • Separated homepage, project pages, and system pages to reflect different user needs
  • Built reusable page structures so future content could be added without redesigning everything
  • Made navigation obvious enough that a new user could understand where to start
Architecture Thinking
  • Project-based structure: annual performance, promotion, templates, and SOP flows
  • Audience-based thinking: operation-oriented HR and system-oriented HRIS need different perspectives
  • Timeline-based pages: users understand not only what exists, but when and why it matters
  • Scalable page hierarchy: each project can keep expanding without breaking the whole site
What this project really reflects

This project is less about front-end development itself and more about system thinking applied to HR operations. The actual value is in translating tacit knowledge into visible structure: deciding how content should be grouped, how users should enter, what sequence supports understanding, and how to reduce dependency on the person who originally built the process.

03 / Impact

What changed because of it

The value of the Hub is not only better documentation. It is the reduction of ambiguity, dependency, and invisible operational logic.

01 —
Clearer Handover
Knowledge became less dependent on verbal explanation and more reusable through structured pages.
02 —
Faster Orientation
New HR members could understand project rhythm, system logic, and template usage more quickly.
03 —
More Scalable Operations
The project created a foundation that can keep expanding as more SOPs, systems, and modules are added.
04 / Reflection

What I learned from building it

This project changed the way I think about HR work. It pushed me further from pure execution and closer to designing systems that other people can reliably operate.

Reflection

Before this project, I was mostly solving issues as they appeared. Building the Knowledge Hub made me think more deeply about how work should be made understandable, transferable, and sustainable. It reinforced my interest in roles that sit between HR operations, systems, and product-like internal design. In other words, I was no longer only asking how to finish the work — I was asking how to design the environment so that the work can continue well even without me.
