01 / Overview

Why this project existed

The Knowledge Hub was not created as a visual exercise. It was created to solve an operational weakness: important know-how was difficult to retain, transfer, and reuse.

Context

In HR work, many processes are highly dependent on experience. Project logic, system settings, file paths, template usage, and execution rhythm often stay fragmented across folders, people, and memory. That makes onboarding slow and handover risky.

Core Problem
  • Project knowledge depended too heavily on individuals
  • SOPs and templates were hard to locate and connect
  • Operational logic was not visible enough for successors
  • Different HR audiences needed different entry points into the same knowledge base
02 / Architecture

How the Hub is structured

The real value of this project is not only the page design itself, but the information architecture behind it: how users enter, how content is grouped, and how the structure stays scalable as more modules are added.

01 —
Home Entry
The homepage acts as the main entry point, introducing the overall knowledge space and guiding users into the right path.
02 —
Project Layer
Project pages organize annual initiatives such as performance, promotion, templates, and timeline-driven execution logic.
03 —
System Layer
System pages provide module-based guidance so users can understand functions, logic, FAQ, and operational use cases.
04 —
SOP / Template Layer
Detail pages break down execution steps, templates, and context so the knowledge is usable rather than only documented.
Architecture Logic
  • Audience-based entry: operational HR and system-oriented HRIS do not consume knowledge in the same way
  • Hierarchy-based grouping: home, project, system, SOP, and template pages each serve a different role
  • Timeline-based reading: users can understand not only what exists, but when and why it matters
  • Scalable structure: new modules can be added without breaking the consistency of the site

04 / Impact

What changed because of it

The value of the Hub is not only better documentation. It is the reduction of ambiguity, dependency, and invisible operational logic.

01 —
Clearer Handover
Knowledge became less dependent on verbal explanation and more reusable through structured pages.
02 —
Faster Orientation
New HR members could understand project rhythm, system logic, and template usage more quickly.
03 —
More Scalable Operations
The project created a foundation that can keep expanding as more SOPs, systems, and modules are added.
05 / Reflection

What I learned from building it

This project changed the way I think about HR work. It pushed me further from pure execution and closer to designing systems that other people can reliably operate.

Reflection

Before this project, I was mostly solving issues as they appeared. Building the Knowledge Hub made me think more deeply about how work should be made understandable, transferable, and sustainable. It reinforced my interest in roles that sit between HR operations, systems, and product-like internal design. I was no longer only asking how to finish the work — I was asking how to design the environment so that the work can continue well even without me.
